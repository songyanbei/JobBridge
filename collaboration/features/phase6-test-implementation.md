## 1. 测试目标

Phase 6 测试的重点，是验证运营后台前端是否能按 Phase 5 API 契约完成真实运营操作，并确认页面交互、路由鉴权、错误处理、审核工作台生产力体验、P0 原型缺陷修复、构建交付能力均满足 MVP 要求。

测试应重点回答以下问题：

- 正式 Vue 前端是否可启动、可构建、可部署
- 登录态、路由守卫、token 失效跳转是否可靠
- 所有 API 是否统一通过 `src/api/request.js` 处理
- 15 个页面是否全部可访问，无白屏
- 审核工作台是否真正支持运营连续审核，而不是只展示列表
- 审核软锁、乐观锁、Undo、驳回理由、键盘快捷键是否与 Phase 5 后端契约一致
- 所有列表页的分页、筛选、排序、导出是否可用
- 所有危险操作是否有二次确认，所有编辑是否有脏数据保护
- `方案设计_v0.1.md` §15.3 的 15 个 P0 原型缺陷是否全部关闭
- 前端构建产物是否可供 Phase 7 Docker / nginx 联调

## 2. 当前现状与测试策略

当前前端是从原型升级为正式 SPA，本阶段测试要覆盖三类风险：

1. **工程风险**：项目能否稳定启动、构建、代理后端、处理刷新和路由
2. **契约风险**：前端调用是否与 Phase 5 API 的路径、参数、响应格式、错误码一致
3. **运营风险**：页面是否能支撑运营高频工作，尤其审核工作台是否具备足够效率和边界处理

测试策略：

- 自动化优先覆盖稳定规则：路由守卫、API 拦截器、通用组件、关键页面冒烟
- 联调重点覆盖真实后端：登录、审核、账号、岗位/简历、配置、报表、日志
- 手工重点覆盖复杂 UX：审核工作台快捷键、锁冲突、驳回面板、Undo、图片预览、脏数据提醒
- 每发现一个后端契约缺口，测试应确认前端已写入 handoff，而不是默默绕过

## 4. 测试环境要求

### 4.1 基础环境

- Node.js 可运行前端工程
- MySQL / Redis / 后端服务已启动
- Phase 5 `/admin/*` API 可访问
- Swagger UI `/docs` 可访问
- 默认管理员账号可用
- Vite dev server 运行在 `http://localhost:5173`
- 后端运行在 `http://localhost:8000`

### 4.2 测试数据

至少准备：

- 管理员 `admin/admin123`
- 待审岗位 3 条：low / mid / high 风险各 1 条
- 待审简历 3 条：含低置信字段、图片附件、超长文本
- 审核锁冲突模拟数据：通过测试夹具、Redis 预置锁或后端 mock 构造不同 `locked_by`；单管理员正常 UI 不要求自然制造冲突
- 已通过 / 已驳回记录各 2 条
- 厂家 5 个、中介 5 个、工人 20 个
- 被封禁用户 2 个
- 在线岗位 10 条，覆盖不同城市、工种、薪资、TTL 状态
- 简历 10 条，覆盖不同性别、年龄、期望城市
- 城市字典、工种字典、敏感词字典种子数据
- system_config 至少包含 `filter.* / llm.provider / rate_limit.*`
- conversation_log 至少 10 条，含 `criteria_snapshot`

### 4.3 工具建议

- 浏览器：Chrome 最新版
- 自动化：Vitest + Vue Test Utils（如开发已接入）
- E2E：Playwright（推荐）或 Cypress
- API 辅助：Swagger UI、curl、Postman
- 数据检查：MySQL client、Redis CLI
- 截图记录：浏览器 DevTools
