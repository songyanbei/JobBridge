## 1. 测试目标

Phase 4 测试的重点，是验证"企微消息从进来到回复"的完整异步链路是否可靠、正确、可观测。

测试应重点回答以下问题：

- webhook 能否在 100ms 内完成验签+幂等+限流+入队并返回 200
- 限流和幂等防线是否真正生效
- Worker 是否能稳定消费队列并完成业务处理
- message_router 是否能正确分流消息类型和意图
- 10 条命令（含 `/人工客服`）是否全部按规格执行
- 出站失败时是否有重试和补偿
- 死信、重试、日志是否可追溯
- 同一用户的消息是否串行处理

## 2. 当前现状与测试策略

当前基础现状决定了本阶段测试策略：

- Phase 2 已测过企微验签/解密/消息解析基础能力
- Phase 3 已测过 7 个业务 service 的核心逻辑
- Phase 4 新增的主要是消息收发链路的编排和异步处理
- 真实企微环境可能不可用（回调公网地址待确认），测试必须能基于 Mock 独立完成

因此本阶段测试重点为：

1. **链路正确性**：webhook → 队列 → Worker → router → 回复
2. **防线有效性**：幂等、限流、分布式锁
3. **异常处理**：重试、死信、出站失败
4. **命令完备性**：10 条命令全覆盖（含 `/人工客服`）
5. **E2E 场景**：三类用户典型流程
6. **可观测性**：inbound_event 状态、日志、队列

## 4. 测试环境要求

### 4.1 基础环境

- MySQL 8.0+：需要 `wecom_inbound_event` 表、`conversation_log` 表、种子数据
- Redis 7+：队列、幂等、限流、session、锁
- Python 3.11+：运行测试和 Worker 进程
- 所有 Phase 1~3 的测试仍然通过

### 4.2 Mock 能力

测试必须能在不依赖真实企微的情况下独立完成，需要以下 Mock：

| Mock 对象 | Mock 方式 | 用途 |
|---|---|---|
| 企微回调 | 构造加密 XML + 签名，POST 到 `/webhook/wecom` | 模拟企微推送消息 |
| `WeComClient.send_text()` | Mock 返回成功或特定错误码 | 模拟出站发送 |
| `WeComClient.download_media()` | Mock 返回图片二进制 | 模拟图片下载 |
| LLM provider | 使用 Phase 3 已有的 Mock provider | 模拟意图识别和 rerank |

### 4.3 测试数据准备

- 管理员账号（seed 数据）
- 预注册厂家用户（至少 1 个）
- 预注册中介用户（至少 1 个）
- 已发布岗位（至少 5 个，覆盖不同城市/工种）
- 已提交简历（至少 3 个）
- 工人用户留空（由测试流程自动注册）

## 5. 测试用例设计

### 5.1 Webhook 回调链路测试

#### TC-4.1.1 验签成功

- **前置条件**：正确的 token、timestamp、nonce、加密消息体
- **操作**：POST `/webhook/wecom` 带正确签名参数
- **预期**：返回 HTTP 200 "success"

#### TC-4.1.2 验签失败

- **前置条件**：错误的 msg_signature
- **操作**：POST `/webhook/wecom` 带错误签名
- **预期**：返回 HTTP 403

#### TC-4.1.3 解密失败

- **前置条件**：签名正确但消息体损坏
- **操作**：POST `/webhook/wecom`
- **预期**：返回 HTTP 200（不触发企微重试），error log 记录

#### TC-4.1.4 幂等 - 首次消息

- **前置条件**：全新 MsgId
- **操作**：POST 消息
- **预期**：Redis 中 `msg:{MsgId}` 被设置，`queue:incoming` 长度 +1，`wecom_inbound_event` 新增一条 `received`

#### TC-4.1.5 幂等 - 重复消息

- **前置条件**：同一 MsgId 已发送过
- **操作**：再次 POST 同一消息
- **预期**：直接返回 200，`queue:incoming` 长度不变，不新增 `wecom_inbound_event`

#### TC-4.1.6 限流 - 正常频率

- **前置条件**：同一用户 10 秒内发送 ≤ 5 条
- **操作**：依次发送 5 条消息
- **预期**：5 条全部入队

#### TC-4.1.7 限流 - 超频

- **前置条件**：同一用户 10 秒内发送第 6 条
- **操作**：快速发送 6 条消息
- **预期**：前 5 条入队，第 6 条不入队、不写 inbound_event，返回 200

#### TC-4.1.8 限流 - 不同用户不互相影响

- **前置条件**：用户 A 已被限流
- **操作**：用户 B 发送消息
- **预期**：用户 B 正常入队

#### TC-4.1.9 响应时间

- **操作**：发送正常消息
- **预期**：响应时间 < 100ms

#### TC-4.1.10 GET 验证端点

- **操作**：GET `/webhook/wecom` 带 echostr 参数
- **预期**：返回解密后的 echostr 明文

### 5.2 Worker 消费与处理测试

#### TC-4.2.1 正常消费

- **前置条件**：Worker 运行中，`queue:incoming` 有消息
- **操作**：等待 Worker 消费
- **预期**：消息被处理，`wecom_inbound_event` 状态从 `received` → `processing` → `done`

#### TC-4.2.2 inbound_event 状态流转

- **前置条件**：发送一条正常消息
- **操作**：追踪 inbound_event 记录
- **预期**：
  - 初始状态 `received`
  - Worker 取出后更新为 `processing`，记录 `worker_started_at`
  - 处理完成后更新为 `done`，记录 `worker_finished_at`

#### TC-4.2.3 userid 分布式锁

- **前置条件**：同一用户连续发送 2 条消息
- **操作**：两条消息进入队列
- **预期**：第一条处理完成后才开始第二条，两条结果互不干扰

#### TC-4.2.4 错误重试 - 第 1 次

- **前置条件**：Mock message_router 第一次抛异常
- **操作**：发送消息
- **预期**：消息重入 `queue:incoming`，`retry_count=1`，`status=failed`

#### TC-4.2.5 错误重试 - 第 2 次后进死信

- **前置条件**：Mock message_router 连续抛异常
- **操作**：消息经过 2 次重试仍失败
- **预期**：消息进入 `queue:dead_letter`，`status=dead_letter`，用户收到"系统繁忙"回复

#### TC-4.2.6 心跳上报

- **前置条件**：Worker 运行中
- **操作**：等待 60+ 秒
- **预期**：Redis 中存在 `worker:heartbeat:{pid}` key，TTL > 0

#### TC-4.2.7 启动自检恢复

- **前置条件**：手动在 DB 中插入一条 `status=processing` 的 inbound_event
- **操作**：启动 Worker
- **预期**：该消息被重新入队处理

#### TC-4.2.8 优雅退出

- **前置条件**：Worker 正在处理消息
- **操作**：发送 SIGTERM
- **预期**：当前消息处理完成后 Worker 退出，不丢消息

## 6. 验收确认项

以下为本阶段测试通过的最低标准：

- [ ] webhook 可稳定接收消息并 < 100ms 返回
- [ ] 限流生效（同一用户 10s 内 > 5 条被限流）
- [ ] 幂等生效（重复 MsgId 不重复处理）
- [ ] inbound_event 状态流转正确
- [ ] Worker 以独立进程运行
- [ ] Worker 心跳上报正常
- [ ] Worker 启动自检恢复正常
- [ ] userid 消息串行处理
- [ ] 10 条命令全部可正确执行（含 `/人工客服`）
- [ ] 3 类用户典型流程 E2E 通过
- [ ] 语音/文件消息返回不支持提示
- [ ] 图片消息留存成功
- [ ] 对话日志入站含 wecom_msg_id，出站为 NULL，无 UNIQUE 冲突
- [ ] 出站失败有重试
- [ ] 死信可追溯
- [ ] docker-compose worker 可正常启停
- [ ] 无同步阻塞式 webhook

### 7.1 模拟企微回调脚本

测试同学应获得开发提供的模拟脚本（或自行编写），能够：

### 7.2 队列状态检查

```bash
# 查看队列长度
redis-cli LLEN queue:incoming
redis-cli LLEN queue:dead_letter
redis-cli LLEN queue:send_retry

# 查看 Worker 心跳
redis-cli KEYS "worker:heartbeat:*"

# 查看幂等 key
redis-cli KEYS "msg:*"

# 查看限流计数
redis-cli GET "rate:{userid}"
```

### 7.3 DB 状态检查

```sql
-- inbound_event 状态分布
SELECT status, COUNT(*) FROM wecom_inbound_event GROUP BY status;

-- 最近的 inbound_event
SELECT * FROM wecom_inbound_event ORDER BY created_at DESC LIMIT 10;

-- 对话日志（验证 wecom_msg_id）
SELECT * FROM conversation_log ORDER BY created_at DESC LIMIT 10;

-- 审计日志（验证出站失败记录）
SELECT * FROM audit_log WHERE action LIKE '%send_failed%' ORDER BY created_at DESC;
```
