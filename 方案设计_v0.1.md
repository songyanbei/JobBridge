## 1. 项目背景与目标

客户基于企业微信运营"厂家—中介—工人"三方招聘撮合服务。目前靠人工在微信里收发信息，随着规模扩大人力已饱和。目标是引入大语言模型，把**信息收集、结构化、匹配推荐、多轮对话**这条链路自动化，以企业微信作为唯一用户入口。

### 1.1 目标
- 自动接收并结构化厂家/中介/工人通过企业微信提交的信息
- 支持多轮对话式检索匹配，返回自然语言推荐结果
- 按角色做字段级权限过滤
- 早期（简版）支撑 5000 用户、约 1600 次/天检索

### 1.2 非目标（早期不做）
- 不做支付、结算、合同等交易闭环
- 不做岗位/简历的详情页（复用客户已有小程序）
- 不做多租户隔离（单池模式）
- 不做 Web / APP 客户端

## 2. 角色与典型场景

### 2.1 四类角色

| 角色 | 入口 | 主要动作 | 可见敏感字段 |
|---|---|---|---|
| 厂家人事 | 企微外部联系人 | 发布岗位、检索工人 | ✅ |
| 劳务中介 | 企微外部联系人 | 双向：发布岗位 / 检索工人 / 帮工人找岗位 | ✅ |
| 工人 | 企微外部联系人 | 提交求职意向、检索岗位 | ❌ |
| 平台运营 | 运营后台 Web | 审核、预注册、运营管理 | ✅（全权） |

### 2.2 典型场景
- **工人检索**：对话描述需求 → 推荐 Top 3 → 修正追问 → 再推荐（多轮循环）
- **厂家检索**：同上，检索对象改为工人
- **中介检索**：双向，系统根据意图自动判断本轮方向
- **上传**：任一角色通过企微发送**文字描述**，系统自动结构化入库（一期仅支持文字，图片作为附件留存但不参与抽取；如需发布含图片的岗位详情，请编辑小程序页面后附链接）

### 2.3 首次交互流程

#### 工人（自动注册）
1. 工人首次发送任意消息 → 系统自动创建 `user` 记录（`role=worker`）
2. 发送欢迎语 + 示例引导：

> "您好！我是【平台名】智能招聘助手
> 我可以帮您找到合适的工作！
>
> 请直接告诉我您想找什么工作，比如：
> · 在苏州找电子厂，5000以上
> · 想找包吃住的，做长期
> · 昆山有没有服装厂的活
>
> 说得越具体，推荐越准确！"

3. 等待工人输入 → 进入 §10 匹配引擎正常流程
4. 后续回归（session 过期后再来）不再发欢迎语，直接进入业务对话

#### 厂家（已预注册）
1. 预注册厂家首次发消息 → 系统识别 `role=factory` 且 `last_active_at IS NULL`
2. 发送个性化欢迎语（含公司名 + 联系人姓名）：

> "您好，xxx公司的xxx经理！欢迎使用【平台名】，您的企业账号已开通。
>
> 您可以：
> · 发送岗位信息（文字描述即可，我会自动整理入库）
> · 直接告诉我招人需求，比如'要招20个普工，苏州，5000以上'"

#### 中介（已预注册）
同厂家流程，欢迎语额外说明双向能力和切换指令。

#### 异常情况
| 场景 | 处理 |
|---|---|
| 未预注册用户说"我要招人" | 按工人注册，回复引导联系客服开通企业账号 |
| 已封禁用户发消息 | 回复"您的账号已被限制使用，如有疑问请联系客服 xxx" |
| 用户只发表情 / "你好" | 回复引导语"请告诉我您的求职需求，比如…" |
| LLM 抽取失败 / 不理解 | 回复"没太理解您的意思，请再具体描述一下需求" |
| LLM 服务完全不可用 | 回复"系统正在维护，请稍后再试" + 运营后台告警 |

## 3. 非功能性需求

| 维度 | 目标 |
|---|---|
| 用户规模 | 5000 注册用户 |
| 检索频次 | 约 1600 次/天（1/3 日活） |
| 单条上传限制 | 文字 ≤2000 字；图片 ≤5 张，单张 ≤1MB |
| 上传排队延迟 | 可接受（分钟级） |
| 检索端到端延迟 | **≤5 秒**（2 次 LLM 调用各 1-2 秒 + SQL <100ms + 网络开销） |
| 数据 TTL | 岗位 30 天 / 简历 30 天 / 对话历史 30 天 |
| LLM API 日预算 | **30-80 元/天**（客户自付，按 Qwen-turbo + Qwen-plus 估算） |
| 可用性目标 | **99%（月度）**，单机部署不做冗余，生产环境另议 |

## 4. 整体架构

### 4.1 分层
沿用 `recruitment_platform_full_flow_cn_v3.svg` 的九层结构，技术组件映射如下：

| 层 | 组件 | 备注 |
|---|---|---|
| 数据输入层 | 企微外部联系人 + 客服账号 | 唯一入口 |
| 接入层 | 企微消息回调服务 | 接收 / 发送消息 |
| 消息处理层 | 消息解析、内容审核 | 角色识别、文字/图片分流 |
| 结构化层 | LLM（意图抽取档） | 上传→JSON Schema，检索→查询条件 |
| 数据存储层 | MySQL（持久数据）+ Redis（会话状态）+ 对象存储 | **不引入向量数据库** |
| 匹配层 | SQL 硬过滤 + LLM（重排生成档） | 轻量方案 |
| 权限过滤网关 | 字段级过滤中间件 | 按 role 输出不同视图 |
| 回复生成层 | LLM（重排生成档）+ 富媒体拼装 | 附小程序详情链接 |
| 输出层 | 企微消息下发 API | |
| 运维层 | 定时任务（TTL / 日报）+ 运营后台（`/admin/*` 路由，见 §13） | 与业务服务同进程部署 |

### 4.2 外部依赖
- **企业微信 API**：消息接收回调、消息发送、外部联系人信息
- **LLM 服务**：通过抽象层调用（见 §4.3），不绑定具体供应商
- **对象存储**：**一期本地文件系统**（`uploads/` 目录，几万张图 ≤ 几十 GB，单机足够；后期迁移 MinIO / OSS 只需改 provider 配置）
- **客户小程序**：承载岗位 / 简历的图文视频详情页

### 4.3 LLM 能力抽象层

LLM 在本系统中承担两类职责，对外暴露为两个抽象接口，**业务代码只依赖接口，不关心具体供应商**：

| 接口 | 职责 | 调用场景 | 性能要求 |
|---|---|---|---|
| `IntentExtractor`（意图抽取档） | 把自由文本解析成结构化 JSON | 上传入库、检索条件抽取、criteria patch 生成 | 便宜、快，准确率够用即可 |
| `Reranker`（重排生成档） | 候选集排序 + 自然语言回复生成 | 匹配引擎第 3 步 | 质量高，上下文窗口 ≥8K |

**候选供应商**（任选一，后期可切换）：

| 供应商 | 意图抽取档 | 重排生成档 | 备注 |
|---|---|---|---|
| 豆包（火山方舟） | 豆包 lite | 豆包 pro | 国内稳定，价格有竞争力 |
| 通义千问（阿里） | Qwen-turbo / Qwen-flash | Qwen-plus / Qwen-max | 生态成熟，支持开源自部署 |
| 开源自部署 | Qwen2.5-7B / Qwen3-4B | Qwen2.5-32B / Qwen3-14B | 数据完全不出境，适合敏感场景 |

**切换成本**：只需改配置文件 + 对应 SDK 封装类，不改业务代码。

**一期选型原则**：先用云 API（豆包或 Qwen 均可），哪个开通快用哪个；积累三个月真实调用量数据后，根据成本/质量/合规需求再决定是否切换或自部署。

**与客户沟通口径**：本方案不绑定特定大模型品牌，具体使用哪家由技术团队在上线前确定，用户侧体验完全一致。

### 4.4 被评估并否决的架构选型

#### LLM Wiki（Karpathy "第二大脑"方案）—— 不采用

**是什么**：三层架构 raw/ → wiki/（LLM 生成维护的 Markdown 知识页）→ schema，通过 Ingest / Query / Lint 三个操作让 LLM 持续沉淀知识，定位为"对抗 RAG 每次从零检索"的知识复利方案。

**为什么不适合本项目**：
1. **使用场景根本不同**：LLM Wiki 针对**长期积累的稳定知识**（论文/笔记/文档），本项目是**短周期事务型数据**（30 天 TTL、每天 100-200 条新增、随时删改）
2. **权限过滤无法实现**：LLM Wiki 输出是 LLM 合成的自然语言段落，无法做字段级屏蔽（§6 的核心需求"工人看不到电话"直接失效）
3. **溯源和审计链断裂**：多条信息融合进实体页后原始记录边界消失，纠纷无法对账
4. **持续 LLM 成本高**：Ingest 要 LLM、Lint 要 LLM、Query 要 LLM，一条岗位入库需 5+ 次 LLM 调用（对比当前方案每次检索仅 2 次）
5. **漂移风险**：LLM 维护的 wiki 会慢慢出现前后矛盾，对交易型系统不可接受

**可能的边角应用**（二期锦上添花，非核心路径）：
- 平台内部运营知识沉淀（行情、规律）
- 客服 FAQ 机器人（如果将来加）

**决策**：核心匹配流程继续按 §10 三步漏斗走，LLM Wiki 思路作为二期可选项备案，与厂家/工人撮合流程完全隔离。

## 7. 结构化字段清单

### 7.0 分层原则

所有字段分三层存储，避免把"软性偏好"硬过滤掉导致 0 召回：

| 层 | 作用 | 匹配时机 | 设计原则 |
|---|---|---|---|
| 🔴 **硬过滤** | MySQL `WHERE` 子句 | 零容忍 deal-breaker | 少而精，宁可漏也别滥 |
| 🟡 **软匹配** | 交给 `Reranker` 打分时参考 | 有容忍度 | 多多益善，能抽就抽 |
| 📄 **原始描述** | LLM rerank 时兜底 | 捕获任何漏抽的信息 | 全量保留 |

硬过滤字段只放三类：违反了完全没意义的、数字边界清晰的、用户明确说"不接受"的。其他能结构化的都放软层。

每张表额外保留 **`extra: JSON`** 扩展字段，前期没想到的新字段先往这里塞，避免改表。三个月后沉淀到一定量再决定哪些 key 提升为正式字段。

### 7.3 关键字典

|---|---|---|
| **城市字典** | 全国 ~340 个地级市 + 直辖市 | 写死在代码或配置，按国家统计局行政区划 |
| **工种大类字典**（一期） | ~10 类：电子厂 / 服装厂 / 食品厂 / 物流仓储 / 餐饮 / 保洁 / 保安 / 技工 / 普工 / 其他 | 写死配置，支持后续扩充二级子类 |
| **计薪方式字典** | 月薪 / 时薪 / 计件 | 写死 |

**归一化要求**：
- 城市：LLM 抽出后必须能映射到字典中的唯一地级市名（"苏州"、"苏州市"、"姑苏区" → `苏州市`）
- 工种大类：LLM 抽出后必须落在字典内，落不上的记为 `其他` 并在 `job_sub_category` 或 `extra` 里保留原描述
- 工种子类：一期**不做字典**，以字符串存 `job_sub_category`，积累三个月后再决定是否升级为二级字典

### 7.4 薪资归一规则

所有薪资统一换算到 **"月综合收入下限"**（单位：元）作为硬过滤依据：

| 原始表达 | 归一规则 | 示例 |
|---|---|---|
| "月综合 5500-6500" | 取下限 | `5500` |
| "底薪 2500 + 加班 + 绩效" | LLM 结合工时/加班估算，**保守取下限** | `4500`（如 LLM 估不出则回退 raw_text） |
| "时薪 18" | `时薪 × 250`（按 22 天 × 约 11 小时保守估） | `4500` |
| "计件月综合 6-8K" | 取下限 | `6000` |

**保守原则**：归一有歧义时取**更低**的值，避免厂家实际给不到但被系统推出去伤害工人信任。

LLM 抽取时把**完整原始薪资描述**保留在 `extra.raw_salary_text`，万一归一算错可以人工追溯。

### 7.5 敏感字段处理原则

性别要求、年龄区间、民族偏好这类字段在法律上是歧视风险点，但蓝领市场现实就是厂家会明确提出。处理方式：

1. **如实抽取**：厂家说"只要女工 18-38"就原样入库，不做平台层面的"强行去敏感化"
2. **运营后台留开关**：`ENABLE_GENDER_FILTER` / `ENABLE_AGE_FILTER` / `ENABLE_ETHNICITY_FILTER`，如果将来监管要求收紧可一键关闭该字段的硬过滤
3. **日志留痕**：所有使用敏感字段的过滤都记入查询日志，便于合规审计
4. **工人展示侧脱敏**：工人看到的岗位摘要**不显示**"只要女工"这类限制词（避免直接暴露歧视文案），但后台过滤时仍生效

### 7.6 字段清单演进策略

- **每张表预留 `extra: JSON`**：前期没覆盖到的新字段先进 extra，不改表
- **工种子类延后字典化**：积累三个月真实数据后决定二级字典
- **硬过滤字段数量**：一期按 §7.1 / §7.2 列的 8 个左右，后续只在**有明确业务证据**时新增（例如 0 召回率过高或业务反馈强烈）
- **字段新增审批**：任何新增硬过滤字段必须经过 0 召回回归测试，防止盲目加字段导致召回率崩塌

### 7.7 敏感个人信息合规约束

本系统采集的字段中，以下属于《个人信息保护法》（PIPL）定义的**敏感个人信息**：

| 字段 | 敏感级别 | 所属表 | 法律依据 |
|---|---|---|---|
| `phone`（电话） | 高 | user / 推荐结果展示 | 个人通信联络信息 |
| `ethnicity`（民族） | 高 | resume | 明确属于敏感个人信息（PIPL 第 28 条） |
| `has_health_certificate`（健康证） | 中 | resume | 间接关联健康状况 |
| `has_tattoo`（纹身） | 中 | resume | 身体特征信息 |
| `taboo`（禁忌/慢性病） | 高 | resume | 健康医疗信息 |
| `gender` + `age` | 中 | resume | 组合后可定位个人 |
| `gender_required` + `age_min/max` + `accept_minority` | 中 | job | 歧视性筛选条件 |

#### 7.7.1 告知与同意

- **工人首次注册时**（§2.3 欢迎语之后），追加一条隐私告知消息：

> "为了给您精准推荐岗位，我们会收集您的求职信息（包括期望城市、工种、薪资、性别、年龄等）。这些信息将用于岗位匹配，并可能被招聘方查看。您发送信息即视为同意。详细隐私政策请查看：[小程序隐私页链接]"

- **厂家/中介发布岗位时**，不需要额外告知（发布信息属于商业信息而非个人信息）
- 用户回复 `/删除我的信息` 时，系统须能执行数据删除（PIPL 第 47 条，被遗忘权）

#### 7.7.2 最小化采集原则

| 字段 | 是否必填 | 合规策略 |
|---|---|---|
| `ethnicity` | **否**，软匹配字段 | 用户不主动提及就不抽取，LLM prompt 不主动追问 |
| `has_tattoo` | **否** | 同上 |
| `taboo` | **否** | 同上 |
| `has_health_certificate` | **否** | 同上 |
| `gender` / `age` | 是（简历硬过滤） | 必须采集，但属于最小必要范围（匹配必需） |
| `phone` | 否（工人可以不提供） | 工人侧不强制采集，厂家侧预注册时填写 |

**LLM prompt 约束**：`IntentExtractor` 的 prompt 中**不主动追问**民族、纹身、健康证、禁忌，只有用户自己提到时才抽取。必填追问仅限 §10.4 定义的字段。

#### 7.7.3 存储安全

| 措施 | 实施方式 |
|---|---|
| **传输加密** | HTTPS（nginx TLS） |
| **敏感字段数据库加密** | `phone` 字段使用 AES-256 加密存储，查询时解密（一期可先明文，二期加密） |
| **访问控制** | `phone` 仅在 `permission_service.filter_fields()` 中对厂家/中介角色解密返回 |
| **日志脱敏** | `loguru` 输出中对 `phone` 字段做 `138****5678` 脱敏，不记录完整号码 |
| **数据库访问审计** | 运营后台的每次数据查看操作记入 `audit_log` |

#### 7.7.4 数据生命周期与匿名化

| 数据 | TTL | 过期处理 |
|---|---|---|
| 简历（含敏感字段） | 30 天 | 软删除 → 7 天后硬删除（含 `ethnicity`、`taboo` 等字段） |
| 对话日志 | 30 天 | 硬删除 |
| 审核日志 | **180 天** | 保留较长，用于合规审计，到期后硬删除 |
| 操作日志 | 180 天 | 同上 |
| 用户主动删除 | 立即 | 用户发 `/删除我的信息` → 立即软删除其所有简历 + 对话日志，7 天后硬删除 |

#### 7.7.5 数据导出与被遗忘权

- **导出**：运营后台"简历管理"的导出 CSV 功能，导出时自动记入 `audit_log`（谁在什么时候导出了哪些数据）
- **被遗忘权**：用户发送 `/删除我的信息` 指令 → 系统删除该 userid 的所有简历、对话日志、会话状态 → 回复确认 → `user` 表保留但 `status` 标记为 `deleted`（保留注册记录用于防重复注册）
- **数据不出境**：一期所有数据存储在国内服务器，LLM API 调用的是国内供应商（豆包/Qwen），不涉及跨境传输

| 角色 | 识别方式 | 理由 |

一人不能同时拥有"厂家"和"工人"双角色，如需请换 userid 注册。

## 9. 内容审核

### 9.1 双层漏斗
1. **自动层**（瞬时）：敏感词正则 + LLM 内容安全接口（各主流供应商均提供）
   - 高危（色情/政治/诈骗联系方式）→ 直接拒收 + 通知
   - 清洁 → 直接入匹配池
2. **人工层**（异步）：灰度内容进运营后台队列，一键通过/驳回

### 9.2 规则
- 未通过的条目**不进入召回池**
- 原始内容保留 30 天供申诉

### 9.3 封禁策略（一期默认值）
- **一期不做自动封禁**，全部由运营手动处理
- 运营在审核工作台 / 账号管理中点击"封禁" → 必须填写理由 → 记入 `audit_log`
- 封禁后：用户发任何消息只收到"您的账号已被限制使用，如有疑问请联系客服 xxx"
- 解封：运营手动操作 → 填写解封理由 → 记入 `audit_log`
- **二期可扩展**：基于三个月累积数据制定自动规则（如"7 天内驳回 ≥3 次自动临时封禁 7 天"）

## 10. 匹配引擎

### 10.1 意图分流 + 三步漏斗

用户每条消息首先经过 `IntentExtractor` 判定意图，再分流到对应处理链：

| intent | 含义 | 后续动作 |
|---|---|---|
| `upload_job` | 发布岗位 | 结构化抽取 → 必填检查/追问 → 入库 → 审核 |
| `upload_resume` | 提交简历 | 同上 |
| `search_job` | 找岗位 | 进入检索漏斗 |
| `search_worker` | 找工人 | 进入检索漏斗 |
| `upload_and_search` | 发布 + 顺便检索 | 先入库，再自动检索匹配结果返回 |
| `follow_up` | 多轮追问/修正 | 继续当前会话 criteria merge |
| `show_more` | "更多"/"换一批" | 从已有候选集取下一批 3 条（排除 shown_items） |
| `command` | 指令（/重新找、/删除、续期等） | 执行对应命令 |
| `chitchat` | 闲聊/无业务信息 | 回复引导语 |

**检索漏斗**（`search_job` / `search_worker` / `follow_up`）：

```
用户 query
  ↓
[1] IntentExtractor 抽意图 → 结构化检索条件 JSON
  ↓
[2] MySQL 硬过滤 → 候选集（通常 < 50 条）
  ↓
[3] Reranker 语义重排 + 生成自然语言回复 → Top N
```

### 10.2 设计要点
- **不引入向量数据库**，当前规模下 LLM 重排已足够
- 单次检索只调用 2 次 LLM（IntentExtractor + Reranker）
- MySQL 硬过滤作用在 deal-breaker 字段：城市、薪资下限、工种大类、人数
- 架构具备升级空间：将来岗位破万时可在 §10.1 第 2/3 步之间插入 embedding 粗排
- LLM 供应商可替换（见 §4.3），不影响本层算法

### 10.3 已确认参数
- 首轮推荐条数：**3 条**，附 "还有 N 条，回复'更多'继续查看"
- 候选不足时：**主动放宽 + 明确告知**（薪资下调 10%，扩邻近城市），回复里说明"已适当放宽条件"
- 候选为零时：不放宽，直接回复"暂未找到完全匹配的，建议调整条件"并引导用户修改

### 10.4 信息采集策略：混合式（自由输入 + 必填追问）

用户（上传或检索）提交自由文本后，`IntentExtractor` 抽取结构化字段并检查**硬过滤字段**完整性。

**上传场景必填字段**：
- 岗位上传：`city`、`job_category`、`salary_floor_monthly`、`pay_type`、`headcount`
- 简历上传：`expected_cities`、`expected_job_categories`、`salary_expect_floor_monthly`、`gender`、`age`

**检索场景必填字段**：
- 工人找岗位：`city`（至少一个）、`job_category`（至少一个）
- 厂家找工人：`city`（至少一个）或 `job_category`（至少一个），有一个即可检索

**追问逻辑**：
1. LLM 抽取完毕后，代码层检查必填字段缺失列表
2. 缺 1-2 个：合并到一句追问（"请问工作地点在哪个城市？期望月薪大概多少？"）
3. 缺 3 个以上：引导语（"信息还不够完整，请补充以下内容：①城市 ②工种 ③薪资"）
4. 用户补充后，**merge 到已有 criteria**，再次检查，循环直到必填齐全
5. **最多追问 2 轮**，如果 2 轮后仍不全，以现有条件先检索一次（宽松匹配），边推荐边继续收集

### 10.5 推荐结果展示格式

回复采用**纯文字消息 + 每条附小程序链接**，格式如下：

**工人视角（检索岗位）**：
```
为您找到 3 个匹配岗位：

① 苏州xxx电子厂 | 普工
   💰 5500-6500元/月（月薪，包吃住）
   📍 苏州市吴中区
   🔧 两班倒，做六休一，月休4天
   👉 查看详情：[小程序链接]

② 昆山xxx科技 | 电子组装
   💰 时薪19元（综合约5700/月）
   📍 昆山市开发区
   🔧 白班为主，8人间宿舍
   👉 查看详情：[小程序链接]

③ 苏州xxx食品 | 包装工
   💰 5000-5500元/月（计件）
   📍 苏州市相城区
   🔧 长白班，包吃，住宿自理
   👉 查看详情：[小程序链接]

还有 12 个相关岗位，回复"更多"继续查看
不满意？直接告诉我调整方向，比如"薪资再高点""要包住的"
```

**厂家/中介视角（检索工人）**：
```
为您找到 3 位匹配的求职者：

① 张xx | 男 35岁
   🔧 期望：电子厂/普工，5000+/月
   📍 期望城市：苏州、昆山
   📞 联系电话：138xxxx5678
   💼 经验：3年电子厂流水线

② ...
```

**设计要点**：
- 每条只展示 4-5 个核心字段，不堆信息
- 编号 ①②③ 方便用户后续引用（"第 2 个不错"）
- 底部固定引导语：告诉用户可以继续聊、怎么聊
- 工人侧**不展示电话/地址**（§6 权限过滤）
- 厂家/中介侧**展示电话**
- 每条附小程序详情链接（查看图片/视频）

## 11. 多轮对话状态管理

### 11.1 会话状态
见 §5.4 `conversation_session`。核心字段：
- `search_criteria`：跨轮次**累积 merge**，非覆盖
- `shown_items`：去重清单
- `history`：最近 6 轮对话

### 11.2 criteria 更新策略
每轮把"当前 criteria"作为上下文喂给 `IntentExtractor`，让它输出 **patch 指令**（add / update / remove 字段），由代码层执行合并，避免 LLM 吐完整对象时出现漂移。

### 11.3 示例
- 轮 1 "苏州找电子厂" → `{城市:苏州, 工种:电子厂}`
- 轮 2 "工资要 5000 以上" → merge → `{城市:苏州, 工种:电子厂, 薪资下限:5000}`
- 轮 3 "换常熟吧" → update → `{城市:常熟, 工种:电子厂, 薪资下限:5000}`

### 11.4 超时
30 分钟无交互自动清空 `search_criteria` 与 `shown_items`。用户也可用 `/重新找` 手动重置。

### 11.5 "更多"指令处理
用户回复"更多"/"换一批"/"还有吗"时，系统从当前 session 的完整候选集中取下一批 3 条（排除 `shown_items` 中已展示的），不重新查询。候选集耗尽时回复"已经是所有匹配结果了，要不要调整条件重新搜？"

### 11.6 引用结果时条目已失效
用户说"第 1 个不错，详细说说"，但被引用的岗位/简历已过期或被下架时，回复"该条目已失效，为您重新推荐"并自动触发新一轮检索。

### 11.7 并发消息串行化
同一 userid 的消息处理必须**串行**，通过 Redis 分布式锁（`SETNX session_lock:{userid}` + 短 TTL）保证顺序。防止用户快速连发两条导致 criteria 覆盖丢失。

### 11.8 会话与检索快照设计

#### 11.8.1 完整会话结构（Redis `session:{userid}`）

```json
{
  "role": "worker",
  "current_intent": "search_job",
  "search_criteria": {
    "城市": "苏州市",
    "工种": "电子厂",
    "薪资下限": 5000
  },
  "candidate_snapshot": {
    "candidate_ids": ["job_101","job_102","job_103","job_104","job_105","job_106"],
    "ranking_version": 1,
    "query_digest": "a1b2c3d4e5f6",
    "created_at": "2026-04-10T14:32:00+08:00",
    "expires_at": "2026-04-10T15:02:00+08:00"
  },
  "shown_items": ["job_101","job_102","job_103"],
  "history": [
    {"role":"user","content":"苏州找电子厂"},
    {"role":"assistant","content":"为您找到 3 个匹配岗位..."}
  ],
  "updated_at": "2026-04-10T14:32:05+08:00"
}
```

#### 11.8.2 字段语义

| 字段 | 说明 |
|---|---|
| `candidate_snapshot.candidate_ids` | 本轮检索 Reranker 排序后的**完整候选 ID 列表**（可能 20-50 条） |
| `candidate_snapshot.ranking_version` | 每次重新检索 +1，用于判断 `show_more` 取的是哪一版结果 |
| `candidate_snapshot.query_digest` | `search_criteria` 的 SHA256 前 12 位，criteria 变更时自动失效快照 |
| `candidate_snapshot.created_at` | 快照生成时间 |
| `candidate_snapshot.expires_at` | 快照过期时间（`created_at + 30 分钟`，与 session TTL 一致） |
| `shown_items` | 已展示给用户的 ID 集合，`show_more` 时跳过 |

#### 11.8.3 "更多"操作的精确语义

**复用快照，不重查**：

```
用户说"更多" / "换一批" / "还有吗"
    ↓
conversation_service.get_next_batch(userid):
    1. 读 session.candidate_snapshot
    2. 检查 query_digest 是否与当前 search_criteria 一致
       ├── 不一致（criteria 已变更）→ 丢弃快照，重新检索，生成新快照
       └── 一致 → 继续
    3. 从 candidate_ids 中排除 shown_items → 剩余候选
       ├── 剩余 ≥ 3 → 取前 3 条，追加到 shown_items
       ├── 剩余 1-2 → 取全部，提示"已是全部结果"
       └── 剩余 0 → 回复"已经是所有匹配结果了，要不要调整条件重新搜？"
    4. 对取出的条目调 permission_service.filter_fields() + 格式化回复
```

**为什么复用快照而不重查**：
- 一致性：避免"更多"时排序变了，用户看到重复或遗漏
- 性能：省掉一次 LLM rerank 调用（约 2 秒 + 0.01 元）
- 用户心智：用户期望"更多"是"翻页"而非"重新搜索"

**快照失效条件**：
- `search_criteria` 变了（用户追问修正）→ criteria_patch 导致 query_digest 改变 → 丢弃旧快照
- session 过期（30 分钟无交互）→ 整个 session 包括快照一起清空
- 用户发 `/重新找` 命令 → 清空

## 12. 接入层（企业微信）

### 12.1 入站
- 通过企微"客户联系—消息接收回调"接收外部联系人消息
- 支持类型：**文本、图片**
- **不支持语音**（一期），收到语音消息时回复"暂不支持语音，请发送文字"
- **不支持文件**（Word/Excel 等），收到时回复"暂不支持文件，请直接用文字描述"
- 图片处理：企微回调返回 `media_id` → 调 API 下载临时文件 → 存入本地 `uploads/` → **一期不做 OCR**，图片仅作为附件留存，不参与结构化抽取

### 12.2 出站
- 出站接口**依赖企微认证级别**（§16.1 #11），当前尚未确认。三种候选方案及降级路径：

| 方案 | 企微 API | 前提条件 | 能力 |
|---|---|---|---|
| **A（首选）** | 客户联系 - 发消息给客户 (`POST /cgi-bin/externalcontact/send_msg`) | 企微已认证 + 开通"客户联系"权限 | 外部联系人 1v1 主动发消息，48h 窗口内不限次数 |
| **B（降级）** | 应用消息 - 发消息 (`POST /cgi-bin/message/send`) | 自建应用 + 外部联系人关注了该应用 | 可发文本/图文/小程序卡片，但需用户先关注 |
| **C（兜底）** | 被动回复 - 在回调中直接返回 XML 响应体 | 无额外权限 | 仅能在回调 5 秒内同步返回一条回复，不支持异步 |

**⚠️ 方案 C 与异步处理模型（§12.5）冲突**——如果只能被动回复，则 LLM 调用必须在 5 秒内完成，超时无法回复。**方案 A 是唯一能支撑完整业务的路径**，必须在开发前确认。

- 消息格式：**纯文字**（`text` 类型），见 §10.5 展示格式规范
- 小程序链接以文字 URL 形式内嵌在消息体中

### 12.3 企微消息回复时效窗口

⚠️ 企微对外部联系人消息的回复存在**48 小时窗口限制**：用户发消息后，平台在 48 小时内可以回复；超过 48 小时需等用户再次主动发消息。

**影响与对策**：
| 场景 | 影响 | 对策 |
|---|---|---|
| 异步上传入库（排队可以慢） | 如果处理 >48h 无法推回"入库成功"通知 | 上传队列目标 **≤30 分钟** 处理完，远在窗口内 |
| 到期前 3 天续期提醒 | 如果用户 48h 内没发过消息，提醒推不出去 | 只对 48h 内有过交互的用户推送；其余用户到期静默下架，下次来对话时告知"您的 xx 岗位已过期，需要重新发布吗？" |
| LLM 处理超时（极端） | 2-4 秒应回复，正常不触发窗口 | 无影响 |

### 12.4 限流
- 企微发送消息 API 限制：单应用约 **200 次/分钟**（具体取决于企业认证级别）
- 1600 次/天检索 × 每次 1 条回复 ≈ **平均 1.1 次/分钟**，远低于上限
- 峰值预估（上午 9-11 点集中检索）≈ **10-20 次/分钟**，仍安全

### 12.5 Webhook 异步处理模型

#### 问题
企微回调有 **5 秒超时**：平台必须在 5 秒内返回 HTTP 200，否则企微会重试（最多 3 次）。但 LLM 调用 2-4 秒 + DB 操作，整条链路很容易超过 5 秒。同步处理会导致：
1. 超时 → 企微重试 → 同一消息处理 2-3 次 → 重复入库、重复推荐
2. 高峰期并发 → 线程全被 LLM 阻塞 → 新请求排队 → 雪崩

#### 解决方案：快速 ACK + 异步 Worker

```
企微 POST /webhook/wecom
    │
    ▼
api/webhook.py:
    1. 验签 + 解密 → 解析出 Message 对象
    2. MsgId 幂等检查（Redis SETNX `msg:{MsgId}` TTL=600s）
       ├── 已存在 → 直接返回 200（重复回调，忽略）
       └── 不存在 → 继续
    3. 写入 Redis 消息队列（RPUSH `queue:incoming`）
    4. 立即返回 HTTP 200 "success"（< 100ms）
    │
    ▼
Worker（**独立进程**，见下方部署说明）:
    1. BLPOP `queue:incoming`（阻塞等新消息）
    2. 写入 `wecom_inbound_event` 表状态 → `processing`
    3. 执行 message_router.py 完整流程（意图 → 处理 → 生成回复）
    4. 调用 WeComClient.send_text() 异步回复用户
    5. 写 conversation_log + 更新入站事件表状态 → `done`
    6. 处理异常 → 重试队列 or 死信队列 + 入站事件表状态 → `failed`
```

#### 幂等去重
- **幂等键**：企微回调的 `MsgId` 字段（每条消息唯一）
- **实现**：`SETNX msg:{MsgId} 1 EX 600`（10 分钟内同一 MsgId 只处理一次）
- 如果 Redis 不可用，降级为 MySQL `conversation_log` 表的 `UNIQUE(MsgId)` 约束

#### 重试策略
- Worker 处理失败（LLM 超时 / DB 异常）→ 消息重入队列，最多重试 **2 次**
- 2 次仍失败 → 入死信队列 `queue:dead_letter`，给用户回复"系统繁忙，请稍后再试"
- 死信队列由运营后台定时检查 or 告警通知

#### 对架构文档的影响
- `api/webhook.py` 改为"验签 + 幂等 + 入队 + 快速返回"，**不再直接调 message_router**
- 新增 `services/worker.py`：消费队列，调用 `message_router.process()`
- `core/redis_client.py` 补入队/出队/幂等相关方法

#### Worker 部署方案（已确认）

**选型：独立进程**，不用后台线程也不用 APScheduler。理由：
1. Worker 挂了不影响 webhook 接收（主进程仍可入队）
2. 可独立 scale（`docker compose up --scale worker=N`）
3. 重启时未处理消息仍在 Redis 队列，不会丢失
4. 通过 `wecom_inbound_event` 表恢复 `status=processing` 的消息（Worker crash 后重启自检）

**`docker-compose.prod.yml` 新增 worker 服务**：
```yaml
  worker:
    build: ./backend
    command: python -m app.services.worker
    depends_on:
      - mysql
      - redis
    env_file: .env
    restart: unless-stopped
```

**Worker 监控**：
- **心跳**：Worker 每 60 秒写 Redis `worker:heartbeat:{pid}` TTL=120 秒。定时任务每 3 分钟检查 heartbeat key 是否存在，不存在则写 `audit_log` 告警
- **队列积压告警**：定时任务每分钟检查 `LLEN queue:incoming`，超过 50 条写 `audit_log` 并推送企微群告警
- **死信告警**：`LLEN queue:dead_letter` > 0 时立即告警

#### 用户级限流（防刷）

在 `webhook.py` 幂等检查通过后、入队之前，加一道 **userid 级频率限制**，防止脚本攻击或用户误触高频发送导致 Redis 队列被打满和 LLM 成本失控。

**规则**：
- 同一 `external_userid` **10 秒内最多 5 条消息**
- 超出限额：不入队，直接返回 200（避免企微重试），异步回复"您发送太频繁了，请稍后再试"
- 被限流的消息**不写入** `wecom_inbound_event` 表（不消耗存储）

**实现**（`core/redis_client.py` 补方法）：
```python
def check_rate_limit(userid: str, window: int = 10, max_count: int = 5) -> bool:
    """返回 True 表示允许通过，False 表示被限流。"""
    r = get_redis()
    key = f"rate:{userid}"
    current = r.incr(key)
    if current == 1:
        r.expire(key, window)
    return current <= max_count
```

**参数可配**：`system_config` 表预留 `RATE_LIMIT_WINDOW_SECONDS` 和 `RATE_LIMIT_MAX_COUNT` 两个配置项，运营可在后台调整。

### 12.6 企微接入可靠性设计

#### 12.6.1 入站事件表

虽然主处理走 Redis 队列，但为了**审计可追溯 + 异常排查**，所有入站回调都同时写一张 MySQL 事件表：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | BIGINT PK | 自增 |
| `msg_id` | VARCHAR(64) UNIQUE | 企微消息 ID，幂等键 |
| `from_userid` | VARCHAR(64) | 发送者 external_userid |
| `msg_type` | ENUM | text / image / voice / event |
| `content_brief` | VARCHAR(500) | 消息摘要（文本取前 500 字，图片存 media_id） |
| `status` | ENUM | `received` / `processing` / `done` / `failed` / `dead_letter` |
| `retry_count` | TINYINT | 已重试次数 |
| `worker_started_at` | DATETIME | Worker 开始处理时间 |
| `worker_finished_at` | DATETIME | Worker 处理完成时间 |
| `error_message` | TEXT | 失败原因 |
| `created_at` | DATETIME | 回调到达时间 |

> 此表不在 `schema.sql` 一期 10 张表里，需要在开发阶段通过 Alembic 迁移添加。

#### 12.6.2 幂等保障（三层防御）

| 层 | 机制 | 覆盖场景 |
|---|---|---|
| **L1 Redis** | `SETNX msg:{MsgId} 1 EX 600` | 企微 5 秒超时重试（最常见） |
| **L2 入站事件表** | `UNIQUE(msg_id)` | Redis 不可用时的降级 |
| **L3 业务层** | `conversation_log` 写入前检查最近 60 秒内是否有相同 userid + 相同 content_hash | 极端情况兜底 |

#### 12.6.3 回调重试处理

企微回调重试规则：5 秒内未收到 200 → 间隔 10s/30s/60s 重试共 3 次。

| 场景 | 系统行为 |
|---|---|
| 首次回调 | 正常入队处理 |
| 重试回调（MsgId 已存在于 Redis） | 直接返回 200，**不入队** |
| Redis 宕机 + 重试回调 | 走 L2 入站事件表 UNIQUE 约束拦截 |
| 入站事件表也写失败（极端） | 走 L3 业务层去重，最差情况用户收到两条相同推荐（可接受） |

#### 12.6.4 消息发送失败补偿

出站消息（`WeComClient.send_text()`）可能因网络波动、API 限流、token 过期等原因失败：

| 失败类型 | 处理 |
|---|---|
| 网络超时 / 5xx | **立即重试 1 次**（httpx retry），仍失败则写 `queue:send_retry` 队列，60 秒后 Worker 重试 |
| access_token 过期（errcode=42001） | 自动刷新 token 后重试 |
| API 限流（errcode=45009） | 写入 `queue:send_retry`，**指数退避**（60s / 120s / 300s） |
| 用户不存在 / 已退出（errcode=60111） | 不重试，标记该 userid 为 `inactive` |
| 重试 3 次仍失败 | 放弃发送，入 `audit_log` 记录"消息发送失败"，运营后台可查 |

#### 12.6.5 48 小时窗口内外差异化处理

| 场景 | 窗口内（48h 内有交互） | 窗口外（超过 48h） |
|---|---|---|
| 检索结果回复 | 正常发送 | 不可能出现（用户刚发了消息才触发检索） |
| 上传入库成功通知 | 正常发送 | 若排队超 48h（极不可能），放弃通知，下次用户来时补告 |
| TTL 续期提醒（主动推送） | 发送提醒 | **不发送**，下次用户主动来对话时补提醒："您的 xx 岗位已过期，需要重新发布吗？" |
| 每日日报推送（面向运营群） | 不受 48h 限制（群消息） | 不受限制 |

### 13.7 账号与鉴权

- 一期仅 **1 个运营管理员账号**（用户名 + 密码，配置在环境变量或后台 `admin_user` 表）
- 密码通过 bcrypt 加盐哈希存储
- 登录后颁发 JWT，带在 `Authorization: Bearer xxx` header
- 所有 `/admin/*` 接口必须验证 JWT
- 24 小时过期需重新登录
- **不提供**：注册、找回密码、多账号、权限分级
- **提供改密码**：`PUT /admin/me/password`（首次登录强制改密码 + 头像菜单手动改密码）

### 13.9 审核工作台详细 UX 要求

审核工作台是整个运营后台**每天使用 4-8 小时的核心页面**，UX 设计必须按生产力工具而非管理页面对待。

#### 13.9.1 定位
- **不是**：Excel 式列表翻页挑着审
- **是**：邮件审阅 / 工单队列式的连续卡片流，运营进入"工作流状态"后一条接一条过

#### 13.9.2 布局（单卡片精读模式 + 列表速览模式切换）

**主内容卡片分三块**：
- **A 块 原始内容**：提交者信息条（头像 + 角色 Tag + 历史信誉）+ 原始文本 + 图片 gallery（缩略图）
- **B 块 LLM 抽取结果**：所有结构化字段表单化展示，**可直接编辑**，每字段旁标注 **LLM 置信度**（绿/黄/红点）
- **C 块 审核建议**（⚠️ 当前原型缺失）：LLM 决策建议 / 触发规则列表 / 风险等级 / 相似内容提示 / 提交者 7 天审核历史

**操作区**：三个大按钮 —— 通过 / 驳回 / 稍后，纵向或横向均可，但**视觉优先级必须远高于字段编辑区**。

#### 13.9.3 十条核心 UX 原则（设计师与开发必须对齐）

1. **键盘优先**：每个核心动作有快捷键，运营手可以离开鼠标。建议键位 `P`（通过）/ `R`（驳回）/ `S`（稍后）/ `E`（编辑）/ `U`（Undo）/ `?`（帮助）。**不建议用 Space 做通过**（与浏览器默认翻页冲突）
2. **零点击通过**：按 `P` 立即通过，**不弹确认对话框**，由 Undo 窗口（10-30 秒）兜底误操作
3. **风险可视化分级**：低风险绿色边框、中风险黄色、高风险红色 + 顶部警告条，运营注意力被自动引导
4. **LLM 抽错一眼可见**：低置信度字段整行背景色淡红，运营扫一眼就知道哪里要改
5. **驳回必须带理由**：点驳回 → 展开预设理由下拉（8-10 个高频选项）+ 自由文本框 + 勾选"通知提交者/加入黑名单"，再次确认后才真正驳回
6. **Undo 必须存在**：每个动作都可撤销，10-30 秒窗口
7. **提交者历史内联**：7 天审核记录直接显示在卡片上，点击弹 Drawer 看完整历史（判断是否惯犯）
8. **图片处理**：缩略图 + 全屏预览 Modal（键盘 ← → 切换）
9. **疲劳管理**：每 50 条弹温和提示 "休息一下"，深色模式可选
10. **批量模式**：列表速览模式下可多选批量操作，**硬上限一次 20 条**，批量必须弹确认

#### 13.9.4 后端契约要求（前后端必须对齐）
- **软锁**：前端打开一条审核时调 `POST /admin/audit/{id}/lock`（300 秒 TTL），切换/离开时调 `unlock`。队列列表里已被他人锁定的条目显示"xxx 正在审核"
- **乐观锁**：通过/驳回/编辑请求必须带 `version` 字段，后端校验版本号一致才执行，不一致返回冲突错误
- **Undo**：后端在通过/驳回后将动作暂存 Redis `undo_action:{id}` TTL=30 秒，前端在 30 秒内可调 `POST /admin/audit/{id}/undo` 撤销

#### 13.9.5 必须设计的页面状态（边界情况）
- 队列为空（"今日审核完毕"庆祝态）
- 网络断开（顶部红条警告，按钮置灰）
- 并发冲突（打开时发现已被锁定 → 提示"xxx 正在审核此条目"，可强制接管）
- 版本冲突（提交时 version 不匹配 → 提示"此条目已被修改，请刷新"）
- 超长内容（局部滚动，不滚动整个页面）
- LLM 服务挂了（空抽取结果 + "请人工录入"提示）
- 编辑中未保存切换下一条（自动保存 + Toast）

### 14.4 关键实现决策记录

| # | 决策 | 理由 |
|---|---|---|
| 1 | **同步 SQLAlchemy，不上 async** | 1600 次/天 ≈ 0.02 QPS，异步的复杂度不值得。LLM 调用慢但可用线程池并发。QPS 涨到 10+ 再考虑切换，成本可控 |
| 2 | **`docker-compose.yml` 挂载 `sql/` 到 `/docker-entrypoint-initdb.d`** | MySQL 首次启动自动按文件名顺序执行 `schema → seed → seed_cities_full`，一条命令拉起完整可用环境 |
| 3 | **`seed.sql` 内嵌真实 bcrypt 哈希** | `admin/admin123` 开箱可登录，不需要额外 init 脚本。生产部署前必须替换 |
| 4 | **`/health` 端点真实查 DB** | `SELECT 1` 验证数据库连通，便于部署验证和监控探活 |
| 5 | **CORS 按环境切换** | `is_development` → `allow_origins=["*"]`，生产环境收紧 |
| 6 | **ORM 模型一期用单文件 `app/models.py`** | 50+ 字段分布在 10 个类里总共 500-700 行，单文件更好导航。代码膨胀到一定程度再拆 |
| 7 | **配置通过 `pydantic-settings` 集中读取 .env** | 类型安全，类型错误启动即暴露；其它模块一律 `from app.config import settings` |
| 8 | **所有 DB 会话通过 `get_db()` FastAPI 依赖注入** | 请求结束自动关闭连接，避免连接泄漏 |

### 14.5 部署与运维

#### 14.5.1 部署方式：Docker Compose 打包部署

生产环境复用开发环境的 `docker-compose.yml` 架构，在此基础上增加业务服务容器：

```
docker-compose.prod.yml
```

**与开发环境的区别**：

| 维度 | 开发 | 生产 |
|---|---|---|
| 业务服务 | 本机 `uvicorn --reload` | Docker 容器 `uvicorn --workers 2` |
| MySQL 初始化 | 自动执行 `sql/` 目录 | 首次部署手动导入或 Alembic 迁移 |

| 日志 | stdout（终端可见） | 写文件 + `docker logs` |
| admin 密码 | `admin123` | **部署前必须替换** |
| `.env` | 默认值直连本地 | 填入真实的企微/LLM/OSS 配置 |

#### 14.5.2 数据持久化
- MySQL：Docker named volume `mysql_data`，定期 `mysqldump` 备份到宿主机（cron 每日 3:00）
- Redis：AOF 持久化，named volume `redis_data`
- 上传文件：bind mount 到宿主机目录（如 `/data/jobbridge/uploads/`）
- 日志：bind mount 到宿主机（如 `/data/jobbridge/logs/`）

#### 14.5.3 运维操作
- **启动**：`docker compose -f docker-compose.prod.yml up -d`
- **更新代码**：`docker compose build app && docker compose up -d app`（只重建业务容器，DB/Redis 不受影响）
- **查日志**：`docker logs -f jobbridge-app`
- **备份**：`docker exec jobbridge-mysql mysqldump -u root -p jobbridge > backup_$(date +%Y%m%d).sql`
- **恢复**：`docker exec -i jobbridge-mysql mysql -u root -p jobbridge < backup_xxx.sql`

### 17.2 指令清单

#### 17.2.1 设计原则
- 指令优先级**高于普通自然语言**；如果一句话同时命中指令和检索意图，先执行指令
- 模糊表达可做同义归并，但**最终归并到固定命令集**，避免实现分叉
- 所有改变系统状态的指令都要写入 `conversation_log`，涉及账号/数据变更的同时写 `audit_log`
- 删除类、封禁类、续期类指令都要有**明确成功/失败回复**

#### 17.2.2 一期命令表

| 指令 | 同义表达 | 适用角色 | 功能 | 前置条件 | 成功回复（示例） | 失败回复（示例） | 审计 | 会话影响 |
|---|---|---|---|---|---|---|---|---|
| `/帮助` | 帮助、怎么用、指令 | 全部 | 返回可用指令和示例 | 无 | "您可以这样用我：找工作可直接说需求；常用指令有 /重新找 /帮助 ..." | n/a | 否 | 不变 |
| `/重新找` | 重来、重新搜、清空条件 | 工人 / 厂家 / 中介 | 清空当前 session，重新开始检索 | 有活跃 session | "好的，已帮您清空当前条件。请重新告诉我您的需求。" | "当前没有可清空的搜索条件，您可以直接输入新需求。" | 否 | 清空 `search_criteria`、`candidate_snapshot`、`shown_items` |
| `/找岗位` | 帮我找工作、切到找岗位 | 中介 | 显式切换到找岗位方向 | 当前用户为 `broker` | "已切换到找岗位模式，您可以告诉我工人的求职需求。" | "只有中介账号可以切换双向模式。" | 否 | 设置会话方向为 `search_job` |

| `/招满了` | 招满了、人招够了、满员了 | 厂家 / 中介 | 标记岗位已招满（`delist_reason=filled`） | 用户名下存在在线岗位 | "已标记岗位【xxx】为招满状态。" | "未找到可操作的岗位。" | 是 | 不变 |

#### 17.2.3 续期命令细则
- `/续期` 无参数：默认对**最近一个未过期岗位**操作，如有多个则返回编号列表请用户确认
- `/续期 15` 或 `/续期 30`：分别映射到 `+15 / +30 天`
- 过期已超过 30 天的岗位不允许直接续期，提示"请重新发布"

#### 17.2.4 删除命令细则
- 一期采用**单步触发 + 明确告知**，不增加二次确认，降低用户操作摩擦
- 删除命令执行后：
  - 立即清空 Redis session
  - 立即软删除简历与对话日志
  - `user.status` 标记为 `deleted`
  - 7 天后硬删除敏感数据
- 如后续客诉较多，二期可升级为 `/删除我的信息` → "回复 YES 确认" 的双步流程
