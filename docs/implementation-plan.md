# JobBridge 分阶段开发与测试实施文档

> 基于《方案设计_v0.1.md》与《docs/architecture.md》整理
> 目标：把当前需求设计转成可执行的开发与测试计划，明确每个阶段的范围、边界、禁止项与验收标准

---

## 1. 文档目标

本文件用于指导 JobBridge MVP 的分阶段实施，覆盖：

- 每个阶段要完成的功能
- 每个阶段明确不做的内容
- 每个阶段的测试重点和测试方式
- 每个阶段的进入条件、退出条件和验收标准

本文档默认面向 **MVP 一期**，以“先打通主链路、再补运营能力、最后做联调和上线准备”为基本策略，不在中途引入二期能力或架构升级。

---

## 2. 总体实施原则

### 2.1 实施原则

1. **先骨架、后链路、再后台、最后联调**。优先把数据模型和主链路打稳，再补管理端和运营能力。
2. **每阶段必须有可验证产物**。不接受“代码写了一半但无法证明可用”的阶段性交付。
3. **严格控制范围**。凡是文档明确写“一期不做”的内容，不在任何阶段夹带实现。
4. **先自动化测试核心逻辑，再做端到端联调**。避免把问题集中到最后一轮发现。
5. **企微接入必须异步化**。Webhook 只做验签、幂等、入队和快速返回，禁止同步执行业务全链路。
6. **权限与合规不后补**。字段级权限过滤、删除流程、敏感字段开关必须从主链路阶段开始纳入实现，不允许留到上线前补丁式处理。

### 2.2 一期全局禁止项

以下内容在所有阶段均视为 **明确不做**：

- 不做支付、结算、合同闭环
- 不做独立 Web/App 用户端
- 不做多租户隔离
- 不做向量数据库 / RAG / 知识库路线
- 不做语音识别、文件解析、图片 OCR 抽取
- 不做岗位 headcount 自动递减
- 不做超出文档定义的新角色和新业务流
- 不做未在文档定义的复杂推荐策略，如协同过滤、画像系统、学习排序
- 不做未经确认的合规策略扩展，如实名核验、人脸核验

## 3. 阶段划分总览

| 阶段 | 名称 | 核心目标 | 主要交付 |
|---|---|---|---|
| Phase 0 | 实施准备与基线冻结 | 建立统一基线，锁定 MVP 范围和环境 | 环境、配置、任务拆分、依赖清单 |
| Phase 1 | 数据与后端骨架 | 建立可运行的后端基础结构和数据模型 | ORM、Schema、配置、基础工具 |
| Phase 2 | 核心基础设施适配层 | 打通 LLM、Redis、存储、企微基础封装 | provider 抽象、客户端、队列基础 |
| Phase 3 | 核心业务服务层 | 完成上传、检索、会话、权限、审核核心逻辑 | service 层主能力与单测 |
| Phase 4 | 企微消息主链路 | 打通“收消息到回消息”的异步闭环 | webhook、worker、router、消息日志 |
| Phase 5 | 运营后台后端接口 | 提供 admin API，支撑审核和运营 | `/admin/*` 后端接口 |
| Phase 6 | 运营后台前端 | 落地运营后台页面和前后端联调 | Vue 管理端 MVP |
| Phase 7 | 定时任务、联调与上线验收 | 完成 TTL、报表、端到端验收与上线准备 | E2E、验收报告、上线清单 |

建议执行顺序固定，不跨阶段并行推进关键链路，避免接口和边界频繁变动。

---

## 4. 分阶段开发与测试计划

## 4.1 Phase 0：实施准备与基线冻结

### 本阶段目标

在正式开发前统一基础约束，确保后续开发不因环境、职责或范围漂移反复返工。

### 本阶段要做的功能/工作

- 确认一期版本边界，以 `方案设计_v0.1.md` v0.20 为唯一需求基线
- 确认 `docs/architecture.md` 为唯一代码架构基线
- 明确目录职责：`backend/`、`frontend/`、`docs/`、`nginx/`
- 校验本地开发环境可运行：MySQL、Redis、Python 3.11+、Node、Docker Compose
- 梳理外部依赖清单并标状态：企微权限、企微管理员、回调公网地址、LLM Key、小程序链接规则、隐私政策页
- 拆分开发任务和测试任务，建立阶段负责人
- 准备测试数据样本：
  - 工人求职文本样本
  - 厂家岗位文本样本
  - 中介双向检索样本
  - 敏感字段样本
  - 异常输入样本

### 本阶段边界

- 只做项目准备和基线冻结
- 不开始实现业务逻辑
- 不接受“边做边改需求基线”的方式

### 本阶段不能做

- 不能在此阶段临时新增业务功能
- 不能跳过外部依赖确认就进入企微联调
- 不能先写前端页面再反推接口设计

### 测试与检查

- 环境检查：本地依赖是否可启动
- 文档检查：需求、架构、实施计划三者是否一致
- 依赖检查：外部依赖清单是否有明确状态和责任人

### 验收标准

- 开发环境可以在本地完整启动基础设施
- 外部依赖确认单已生成，且高风险项有备选方案
- 阶段拆分、负责人、测试样本已确定
- 后续开发所依据的文档版本已冻结

## 4.2 Phase 1：数据与后端骨架

### 本阶段目标

把文档里的数据模型、DTO、基础配置和通用能力落成一个可运行的后端骨架，为后续业务服务提供稳定承载。

### 本阶段要做的功能

- 完成 `models.py` 与 `backend/sql/schema.sql` 的一致性映射
- 完成基础 Pydantic schema：
  - `schemas/user.py`
  - `schemas/job.py`
  - `schemas/resume.py`
  - `schemas/conversation.py`
  - `schemas/llm.py`
  - `schemas/admin.py`
- 完成配置与基础工具：
  - `config.py`
  - `db.py`
  - `core/exceptions.py`
  - `core/pagination.py`
  - `core/redis_client.py`
- 明确软删除、JSON 字段、枚举字段的 ORM 实现方式
- 准备种子数据可导入能力：
  - 管理员账号
  - 城市字典
  - 工种字典
  - 敏感词初版

### 本阶段边界

- 只搭建数据骨架和基础模块
- 不实现完整业务编排
- 不接企微，不调真实 LLM 主链路

### 本阶段不能做

- 不能在 service 未设计完成前把业务逻辑塞进 API 层
- 不能绕过 schema 直接让 API 暴露 ORM 对象
- 不能在 ORM 层做全局黑魔法式过滤，统一放在 service 层控制

### 测试重点

- ORM 映射测试：
  - 表字段、类型、默认值、枚举值与 DDL 一致
  - JSON 字段可正确读写
  - `extra` 字段变更可被追踪提交
- 数据库集成测试：
  - 建表成功
  - 种子数据导入成功
  - 基础增删改查成功
- Redis 基础测试：
  - 可连通
  - 基本 set/get/delete 正常
  - TTL 正常

### 验收标准

- `from app.models import *` 可以成功导入
- MySQL 能基于当前 DDL 成功建表
- 核心 schema 可完成基本序列化/反序列化
- Redis 与数据库基础操作自动化测试通过
- 本阶段代码不包含业务“临时兜底”逻辑

## 4.3 Phase 2：核心基础设施适配层

### 本阶段目标

把后续主链路依赖的外部设施抽象和封装稳定下来，做到“业务层只依赖接口，不依赖具体厂商”。

### 本阶段要做的功能

- 完成 LLM 抽象层：
  - `llm/base.py`
  - `llm/prompts.py`
  - `llm/providers/qwen.py`
  - `llm/providers/doubao.py`
  - provider 工厂注册逻辑
- 完成对象存储抽象层：
  - `storage/base.py`
  - `storage/local.py`
- 完成企微基础封装：
  - `wecom/crypto.py`
  - `wecom/callback.py`
  - `wecom/client.py`
- 完成消息队列的基础能力封装：
  - 幂等键规范
  - `queue:incoming`
  - `queue:dead_letter`
- 明确 LLM prompt 版本管理与错误兜底

### 本阶段边界

- 本阶段关注“基础设施适配”
- 不在这里实现完整消息路由编排
- 不在这里实现运营后台接口

### 本阶段不能做

- 不能让 service 直接 import 具体 provider 类
- 不能在 prompt 中输出非严格 JSON
- 不能将真实业务判断散落在 `wecom/` 或 `storage/` 层

### 测试重点

- LLM provider 单元测试：
  - 正常文本可返回符合契约的 `IntentResult`
  - 返回非 JSON 时兜底逻辑生效
  - 超时和重试逻辑生效
- 对象存储测试：
  - 保存、读取 URL、删除正常
  - 键命名符合约定
- 企微基础测试：
  - 验签
  - 消息解密
  - XML 解析
  - 消息发送 API 封装返回值处理
- 队列基础测试：
  - 入队/出队
  - 幂等去重
  - 死信转移

### 验收标准

- LLM 调用可以通过统一工厂切换 provider
- 本地存储可正常保存图片附件
- 企微回调消息可被正确解密并转成统一消息对象
- 队列基础能力可稳定运行
- 所有基础设施模块都已可被 mock，便于后续 service 层测试

## 4.4 Phase 3：核心业务服务层

### 本阶段目标

完成系统最关键的业务能力，让“结构化、检索、会话、审核、权限”这些核心逻辑在不依赖 UI 的情况下可独立运行和测试。

### 本阶段要做的功能

- `user_service.py`
  - 用户识别
  - 工人自动注册
  - 厂家/中介首次欢迎判断
  - 封禁状态判断
- `intent_service.py`
  - 意图分类
  - 结构化抽取
  - 必填字段缺失追问逻辑
- `conversation_service.py`
  - Redis session 读写
  - `criteria_patch` merge
  - `candidate_snapshot` 管理
  - `shown_items` 记录
  - `/重新找` 清空
- `upload_service.py`
  - 岗位/简历入库流程
  - 文字信息抽取
  - 图片仅留存不抽取
  - 审核接入
- `audit_service.py`
  - 敏感词检测
  - LLM 安全检查接入点
  - 待人工审核队列
- `search_service.py`
  - 硬过滤
  - 候选集构造
  - rerank
  - show_more 依赖快照取下一批
- `permission_service.py`
  - 工人视角岗位字段过滤
  - 厂家/中介视角简历字段过滤

### 本阶段边界

- 只完成服务层业务核心
- 不要求企微实链路联通
- 不要求 admin API 全量完成

### 本阶段不能做

- 不能在 search 中引入向量检索
- 不能把敏感字段过滤放到前端做
- 不能把会话状态写回 MySQL 替代 Redis
- 不能让图片参与一期字段抽取
- 不能把“更多/换一批”重新跑一遍全量搜索替代快照机制

### 测试重点

- 单元测试：
  - 意图分类结果映射
  - `criteria_patch` 合并逻辑
  - show_more 逻辑
  - 权限过滤矩阵
  - 必填字段追问逻辑
- 集成测试：
  - 岗位上传入库流程
  - 简历上传入库流程
  - 工人找岗位流程
  - 厂家找工人流程
  - 中介双向流程
- 边界测试：
  - 空输入、纯闲聊、表情、超长文本
  - LLM 解析失败
  - 0 召回
  - 候选条目失效
  - session 过期
- 合规测试：
  - 工人看不到电话和详细地址
  - 敏感过滤开关开启/关闭逻辑正确
  - `/删除我的信息` 相关 service 流程可执行

### 验收标准

- 可用脚本独立模拟完整业务服务流程
- 工人、厂家、中介三类典型流程均可跑通
- `show_more` 基于快照工作正常
- 字段级权限过滤符合文档矩阵
- 结构化抽取、检索、追问、审核四条核心服务链都有自动化测试覆盖

## 4.5 Phase 4：企微消息主链路

### 本阶段目标

把系统最核心的用户入口打通，形成“企微收消息 → 异步处理 → 回复用户”的完整闭环。

### 本阶段要做的功能

- `api/webhook.py`
  - 验签
  - 解密
  - MsgId 幂等检查
  - 入队
  - 快速返回 200
- `services/worker.py`
  - 消费 `queue:incoming`
  - 调用 `message_router.process()`
  - 错误重试
  - 死信处理
- `services/message_router.py`
  - 用户识别
  - 消息类型分流
  - 文本主链路分发
  - 图片留存
  - 语音/文件拒绝回复
  - command / chitchat / upload / search / show_more 分流
- 对话日志写入
- 企微回复发送
- 出站失败补偿基础逻辑

### 本阶段边界

- 只聚焦主消息链路
- 不在此阶段完成所有 admin 后台功能
- 不在此阶段追求复杂运营报表

### 本阶段不能做

- 不能在 webhook 中直接调 `message_router`
- 不能为了“快”而去掉幂等检查
- 不能把失败消息直接吞掉不记录
- 不能支持一期未定义的媒体类型

### 测试重点

- 回调链路测试：
  - 企微回调验签、解密成功
  - 重复消息不会重复处理
  - webhook 返回时间符合要求
- 队列消费测试：
  - worker 可稳定消费
  - 处理异常进入死信
  - 可观察到重试与补偿
- 业务 E2E 测试：
  - 新工人首次消息 → 自动注册 → 欢迎语
  - 工人求职 → 推荐 Top 3
  - 厂家发布岗位 → 入库成功
  - 中介切换 `/找岗位` `/找工人`
  - `/帮助` `/重新找` `/下架` `/招满了` `/删除我的信息`
- 异常测试：
  - 企微消息重复投递
  - LLM 超时
  - 发送失败
  - 图片消息
  - 语音/文件消息

### 验收标准

- webhook 可稳定接收企微消息并在短时间内返回
- 异步 worker 能稳定消费并完成回复
- 主链路典型场景可通过企微联调验证
- 死信、重试、日志链路可观测
- 无同步阻塞式 webhook 实现残留

## 4.6 Phase 5：运营后台后端接口

### 本阶段目标

为审核、配置、账号管理、字典维护和报表能力提供后端 API，支撑运营人员使用后台。

### 本阶段要做的功能

- 登录与鉴权：
  - `/admin/login`
  - `/admin/me`
  - `/admin/me/password`
- 审核工作台：
  - 队列列表
  - 详情
  - lock/unlock
  - pass/reject/edit/undo
- 账号管理：
  - 厂家、中介、工人、黑名单
  - 封禁/解封
- 岗位与简历管理：
  - 列表
  - 详情
  - 编辑
  - 下架/延期
- 字典管理：
  - 城市别名
  - 工种字典
  - 敏感词
- 系统配置：
  - 读取
  - 单项更新
- 基础报表：
  - dashboard
  - trends
  - funnel
  - export
- 对话日志查询与导出
- 小程序点击事件回传接口

### 本阶段边界

- 先做 MVP 后台，不做精细 RBAC
- 默认单管理员模式即可
- 后台重点是“可运营”，不是“复杂 BI 平台”

### 本阶段不能做

- 不能新增文档外的后台业务模块
- 不能把审核锁机制简化掉
- 不能省略乐观锁和 undo 支持
- 不能把运营配置直接写死在代码里绕过配置表

### 测试重点

- API 契约测试：
  - 路径、方法、参数、响应格式与架构文档一致
- 鉴权测试：
  - token 正常
  - token 过期
  - 无权限访问
- 审核工作台测试：
  - 并发锁
  - 版本冲突
  - pass/reject/edit/undo
- 数据管理测试：
  - 筛选
  - 分页
  - 排序
  - 导出
- 事件回传测试：
  - API Key 鉴权
  - 10 分钟去重

### 验收标准

- Swagger 中 admin API 可完整调通
- 审核工作台相关接口能支撑单人审核闭环
- 岗位、简历、账号、字典、配置、日志、报表接口可用
- 所有接口返回格式统一
- 并发修改场景有保护措施

## 4.7 Phase 6：运营后台前端

### 本阶段目标

实现可用的运营后台界面，并完成前后端联调，让运营人员能真正执行审核、查询和配置操作。

- 搭建前端基础：
  - Vue 3 + Vite + Element Plus + Pinia + Vue Router
  - 登录态管理
  - 请求拦截器
  - 路由守卫
- 页面实现：
  - 登录页
  - Dashboard
  - 审核工作台
  - 厂家/中介/工人管理
  - 黑名单
  - 岗位管理
  - 简历管理
  - 字典管理
  - 系统配置
  - 数据看板
  - 对话日志
- 组件实现：
  - Layout
  - PageTable
  - DetailDrawer
  - ImagePreview
- 通用交互：
  - 分页
  - 排序
  - 筛选
  - 导出
  - 二次确认
  - loading
  - Toast
  - 脏数据离开提醒

### 本阶段边界

- 前端只服务运营后台
- 不扩展到用户端产品
- 不做视觉重设计优先级之外的炫技特性

### 本阶段不能做

- 不能在页面里直接散写 axios
- 不能绕过统一响应协议
- 不能把权限控制依赖前端展示隐藏
- 不能开发文档未定义的新页面

### 测试重点

- 前端单元测试：
  - store
  - request 拦截器
  - 通用组件
- 联调测试：
  - 登录与 token 续期/失效
  - 审核工作台全流程
  - 管理页筛选/排序/导出
  - 配置保存
  - 日志查询
- 兼容与交互测试：
  - 常见分辨率
  - loading/空状态/错误状态
  - 键盘快捷键

### 验收标准

- 运营后台核心页面均可正常访问
- 前后端接口联调通过
- 审核、账号管理、岗位简历管理、配置管理、日志查询可完成完整操作闭环
- 前端遵守统一请求封装和统一错误处理规范

## 4.8 Phase 7：定时任务、联调与上线验收

完成系统收尾能力、端到端稳定性验证和上线前准备，输出正式验收结果。

## 8. 实施中的变更控制

实施过程中如需新增或修改需求，建议按以下规则处理：

1. 不影响当前阶段边界的小改动，可作为当前阶段缺陷修复处理。
2. 影响数据模型、接口契约、主链路或验收指标的变更，必须回写到 `方案设计_v0.1.md` 与 `docs/architecture.md` 后再开发。
3. 明确属于二期能力的内容，统一进入 backlog，不插入一期主计划。

## 9. 建议的执行顺序总结

推荐按以下顺序推进：

1. 先完成 Phase 0-1，锁定基线并打稳数据与代码骨架
2. 再完成 Phase 2-3，把业务核心在本地和测试环境中独立跑通
3. 然后完成 Phase 4，尽早验证企微主链路风险
4. 再完成 Phase 5-6，补齐后台可运营能力
5. 最后进入 Phase 7 做收口、验收、上线准备

这样安排的核心原因是：**JobBridge 的真正风险不在页面数量，而在消息异步链路、会话正确性、字段权限和 LLM 结构化质量**。因此实施优先级必须围绕主链路，而不是围绕页面数量。
